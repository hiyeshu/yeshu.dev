> 上下文不是仓库，是一块空地。

仓库给人的感觉是可以无限堆叠，可以分层，可以随时翻找。

但上下文窗口不是这样。它更像一块面积固定的空地。

8K、32K、200K、1M，只是这块地的大小。参数表写多少，它就只有多少。不是你愿意多放，它就会多长出一层楼。

它也是平的。

模型不是走进一栋楼，一层层找资料。它是一次性看见地上所有东西：系统提示、工具定义、memory、历史对话、当前问题、RAG、贴进来的文档。

每一样东西都得落地。

没有谁可以悬空。

所以做 agent 的第一性问题不是「还能塞什么」，而是：

**这块地应该怎么分？**

## 占地是有代价的

地满了，就要处理。

最粗暴的是截断，直接把东西丢掉。模型会失忆。

稍微体面一点的是摘要，把东西收纳起来。信息还在，但质感没了。

再复杂一点的是检索，临时从外面运资料进来。运对了有用，运错了就是白占地。

这些都不是高级策略。

它们是没地方放之后的应急处置。

上下文越满，回答越慢，越贵，也越糊。

地大不等于会用。很多时候，大窗口只是让错误的东西可以被塞得更多。

Lost in the Middle 不是玄学。

一块地太大，模型站在一头看另一头，中间那片本来就容易被忽略。空间变大，只是降低了被挤出去的概率，不代表每个东西都被认真看见。

## 谁在规划这块地

这里有两个角色：建筑师和 build。

建筑师在对话发生之前工作。

他决定 system prompt 怎么写，工具入口开几个，memory 留什么，历史超过几轮开始摘要，RAG 用完要不要带走。

这些东西一旦写下去，agent 的气质就定了大半。

好建筑师会留白。

差建筑师把每一寸都规划上用途，最后第一阵风来，房子就塌。

build 是运行时。

用户按下回车的那一刻，运行时开始施工。它按图纸取材料：系统提示、工具、历史、memory、当前问题、外部文档。然后临时拼成一栋房子，交给模型看一眼。

回答结束，这栋房子就拆了。

下一轮再重盖。

模型并不「记得」上一轮的房子。

所谓记忆，是建筑师规定了下一轮要从相册里取哪几张照片，再重新挂上墙。

顺序也是命运。

同样的材料，先放系统提示再放工具，和反过来，注意力分布完全不同。上下文不是资料列表，而是一个现场。

## 问题通常不在 build

Agent 表现不好时，最容易做的事是改 build。

多塞几轮历史，多召回几段 RAG，多放几条用户画像。

但很多体验问题，根子不在施工现场，而在图纸。

空地一开始没规划好，施工再卖力，也是在错误地基上盖楼。

反过来也成立。

图纸再好，build 偷懒一样会塌。建筑师明明写了「贴入文档用完即走」，但 build 没执行，文档慢慢沉淀进历史，几轮之后地就爆了。

这不是设计问题，是施工问题。

所以做 agent 之前，先问自己一句：

**我现在该当建筑师，还是该优化 build？**

## 这块地是为谁留的

前面都把 agent 当主角。

但真做产品，关系应该反过来。

Agent 是这块地的管家。人是这块地的居民和客人。

系统提示不是为了展示模型多专业，而是让人不用反复交代风格。

工具定义不是为了堆能力，而是让人不用自己动手。

memory 不是为了证明 agent 记性好，而是让人不用每次自我介绍。

历史对话不是档案馆，而是客厅里的相册。

当前问题才是正在说话的人。

RAG 和贴入文档只是临时搬来的参考书，用完就该带走。

这样看，占比就不再是技术参数，而是待客之道。

长期设施要压到极致。

关系记忆只留事实、承诺和偏好。

当前问题要尽量保留原文，因为那是此刻的声音。

临时材料要用完即走，不要沉淀成垃圾。

留白不是浪费。

留白是为了让人进来之后，还有地方坐下。

好的 agent 不是记住一切。

好的 agent 知道，此刻什么值得被摆在桌面上。
